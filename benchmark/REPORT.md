## 3. 根本原因分析

### 🔴 Critical: SigLIPモデルの用途ミスマッチ

**問題:** `prithivMLmods/deepfake-detector-model-v1` はディープフェイク（顔スワップ）検出用に訓練されており、AI生成動画の検出には最適化されていない。

- **ディープフェイク:** 既存の顔映像に別の顔をAIで合成したもの（顔の細部が不自然）
- **AI生成動画:** Runway/Kling/Lumaで1から生成した動画（顔が存在しない場合も多い）

実験結果では本物の顔映像（vlog）のほうが「Fake」スコアが高く出た（mean_real=71.3 vs mean_ai=56.2）。これは旅行vlogには実在する人間の顔が写っており、ディープフェイク検出器が「顔っぽいもの」に反応したためと考えられる。

### 🔴 Critical: 光学フローの仮定が逆

**問題:** 現在の実装は「フロー分散が低い = AI的に滑らか = AI生成」という仮定だが、実際には逆。

- **本物vlog:** 活発なカメラ移動・手ブレ → フロー分散が**高い**
- **AI生成動画デモ:** コントロールされた静的カメラ → フロー分散が**低い**

結果: 本物動画が66.9、AI動画が36.9と、スコアが逆転している。

### 🟡 Important: メタデータ署名が不在

ダウンロードした動画40本でAIツール署名を検出した動画: **0本**

YouTubeはアップロード時にメタデータの多くを削除するため、この手法はYouTube動画では機能しない。直接生成されたファイル（Sora API出力等）でのみ有効。

### 🟡 Important: FFT均一度スコアが機能不全

全動画でFFTスコアが0〜13の範囲に集中（AI: 2.1, Real: 3.4）。閾値55に全く到達しない。

**原因:** 現在の実装は高周波成分の**変動係数(CV)**で測定しているが、動画フレームはどちらのカテゴリでも自然なテクスチャ変動を持つため、均一度の差が現れにくい。より効果的なアプローチはDCT係数の分布分析またはGAN特有のスペクトルアーチファクト検出。

### 🟢 Positive: オーディオ解析は正しい方向

Audio層のみが期待通りの方向性を示した（AI: 24.4 > Real: 12.7）。ただし絶対値が低く、閾値55には到達しない。AI生成動画の多くに音声がなく不確実性が高い。

## 5. 改善提案

### 🔑 Priority 1: SigLIPモデルの交換 (最重要)

ディープフェイク検出モデルをAI生成コンテンツ検出モデルに置き換える：

**推奨モデル候補:**
1. `Hezi-98/CLIP-based-DeepFake-Det` — CLIP使用、AI生成全般に対応
2. `haywire18/Deepfake-Detector-Model-v2` — より新しいモデル
3. `Organika/sdxl-detector` — StableDiffusion生成コンテンツ検出
4. カスタム訓練: Runway/Kling/Luma動画でfine-tuning

### 🔑 Priority 2: 光学フロースコアの反転

現在: `smoothness = 1.0 - cv_flow * 0.5`  
→ 修正: `irregularity = cv_flow * 0.5` (高irregularity = 人間らしい = 低AI確率)

スコア計算を逆転させる。または、フローの「過剰な均一性（AI的）vs 自然なブレ（人間的）」を新しい指標で測定する。

### 🔑 Priority 3: FFT解析の高度化

現在の変動係数(CV)から、GAN/拡散モデル特有のスペクトルアーチファクト検出へ変更：
- Azimuthal average (Durall et al., 2020)の実装
- 高周波スペクトル減衰曲線の傾きを測定
- AI生成画像のピクセル相関パターン検出

### 🔑 Priority 4: Ensembleの再調整

現在の重みは根拠なく設定。最適化すべき方向：

```python
# 修正案（暫定）
OPTIMAL_WEIGHTS = {
    "siglip": 0.30,      # 新モデルで大幅改善期待
    "fft_fixed": 0.20,   # 修正後のFFT
    "optical_flow_inv": 0.25,  # 反転後のOF
    "audio": 0.15,       # 現状で唯一機能している
    "metadata": 0.10,    # 生成APIから直接取得したファイルのみ有効
}
```

グリッドサーチで最適化すべき。

### 🔑 Priority 5: データセット品質向上

- **ディープフェイク動画の収集:** FaceForensics++ 申請 (2〜3週間待ち)
- **AI生成動画の多様性:** Sora API直接出力、Runway API出力など元ファイルを使用
- **動画の統一:** 長さ・解像度を揃える (現在15秒〜3分のmix)
- **顔有無でのセグメント化:** 顔あり/顔なしで別個に評価

## 8. 結論

### 現状の正直な評価

**FakeGuardの現在のアーキテクチャはAI生成動画の検出に対して機能していない。**

- SigLIPモデルはディープフェイク（顔スワップ）用に訓練されており、AI生成動画には転用不可
- AUC-ROC 0.1175はランダム（0.5）を大幅に下回り、実際にはAIを「本物」と判定する傾向がある
- 光学フロー・インターフレーム・FFTの各層も期待した方向に機能していない
- Audio層のみ弱い正の信号を示す

### 何を学んだか

1. **「ディープフェイク検出」≠「AI生成動画検出」** — 全く異なる問題設定
2. AI生成動画の特徴は顔の不自然さではなく、テンポラル一貫性・素材感・音声の均一性にある
3. 実際のユーザー（旅行vlog）との区別は光学フローでは困難
4. メタデータ手法はプラットフォーム経由の動画には機能しない
